# Other Modern Python Tools

This page records useful tools that are not part of the base template yet, or
are not wired into the default gates. Treat it as an adaptation guide for future
domain-specific forks, not as a backlog to install by default.

| What it is | Why and when you need it | Why it is not part of this template so far |
| --- | --- | --- |
| [tox](https://github.com/tox-dev/tox) - virtual environment and test automation for Python projects. Use [tox-uv](https://github.com/tox-dev/tox-uv) if adopting tox in this repo. | Use it for real interpreter/dependency matrices, package install checks, or CI parity across several Python versions. | The template currently targets one pinned Python version, has no package build target, and already routes local plus CI commands through `uv` and `mise`. Adding tox now would duplicate orchestration. |
| [nox](https://github.com/wntrblm/nox) - Python-scripted automation sessions for tests, linting, release, and matrix-style workflows. | Use it when checks need custom Python control flow, generated sessions, or richer conditional logic than a task runner should carry. | `mise.toml` is enough for the current fixed set of commands. There is no custom session logic yet. |
| [Hatch](https://github.com/pypa/hatch) - Python project manager and build backend. | Use it when this stops being a virtual dependency-only template and becomes a distributable package with wheels, sdists, versioning, or publishing. | `pyproject.toml` intentionally has no `[build-system]`; `uv` treats this as a virtual project until `src/` gains a real package. |
| [PDM](https://github.com/pdm-project/pdm) - Python package and dependency manager with lockfile and build workflows. | Use it if a downstream project or team is already standardized on PDM and wants that workflow end to end. | The base template already standardizes dependency resolution, locking, and command execution on `uv`. A second project manager would add competing lock and install paths. |
| [Poetry](https://github.com/python-poetry/poetry) - Python dependency management and packaging workflow. | Use it when an organization requires Poetry for publishing, dependency groups, or established internal workflows. | `uv` is the selected default here. Poetry would be a parallel packaging and locking model rather than an incremental guardrail. |
| [deptry](https://github.com/fpgmaas/deptry) - dependency health checker for missing, unused, and transitive dependencies. | Use it once the project has real imports and dependencies, especially for libraries and services where dependency drift matters. | The current template has an empty runtime dependency set. A side branch has experimented with `deptry`, but it still needs low-noise defaults for sparse `src/` trees before it belongs on `main`. |
| [pip-audit](https://github.com/pypa/pip-audit) - PyPA dependency vulnerability scanner. | Use it when you need PyPA-specific audit output, policy integration, or a second opinion beside the package manager's native audit command. | The current template already has `uv audit` in the lint suite. Previous validation also found `pip-audit` was not installed in this repo. |
| [OSV-Scanner](https://github.com/google/osv-scanner) - standalone vulnerability scanner backed by the Open Source Vulnerabilities database. | Use it when a project wants audit output independent of `uv`, or when security tooling is standardized across multiple ecosystems. | `uv audit` is already the default dependency-vulnerability gate. A separate scanner should be added only when its reporting or policy integration is needed. |
| [Dependency Review Action](https://github.com/actions/dependency-review-action) - GitHub Action for reviewing dependency changes in pull requests. | Use it when pull requests should show dependency license, vulnerability, or supply-chain risk before merge. | It is GitHub-specific and policy-heavy. The template already has Dependabot, Renovate, and `uv audit`, so this belongs in repos with explicit dependency-review rules. |
| [Semgrep](https://github.com/semgrep/semgrep) - rule-based static analysis for security, correctness, and organization-specific code patterns. | Use it for domain-specific rules, compliance scans, framework misuse checks, or patterns that Ruff/Bandit do not express well. | It needs owned rules and false-positive triage. The generic template already includes broad linting, Ruff security rules, GitHub Actions scanning, and secret scanning. |
| [mypy](https://github.com/python/mypy) - mature static type checker with broad ecosystem and plugin support. | Use it when a project needs stable type checking semantics, framework plugins, or compatibility with an existing mypy-based codebase. | The template currently uses `ty` for fast baseline type checking. Running mypy too early would duplicate the type gate before the project has real typing needs. |
| [Pyright](https://github.com/microsoft/pyright) - fast static type checker from Microsoft, commonly used through Pylance and editor workflows. | Use it when the team wants Pyright/Pylance semantics, strict editor parity, or a TypeScript-adjacent type checking model. | `ty` is the current default type checker. Pyright should be chosen deliberately when a downstream project wants its diagnostics and configuration model. |
| [Pylint](https://github.com/pylint-dev/pylint) - mature Python static analyzer with broad diagnostics and plugin support. | Use it when a downstream project wants Pylint-specific checks, framework plugins, or compatibility with an existing Pylint policy. | Ruff already covers the selected Pylint rule families through `PLC`, `PLE`, `PLR`, and `PLW`. Running both by default would add overlap before the project has code that justifies it. |
| [Python Semantic Release](https://github.com/python-semantic-release/python-semantic-release) - automated versioning and release publishing from commit history. | Use it once the project has a real release policy and wants tags, changelogs, and package publishing derived from Conventional Commits. | The template does not yet define a release cadence or package publication flow. Adding release automation now would create policy before product shape. |
| [Towncrier](https://github.com/twisted/towncrier) - changelog-fragment workflow for Python projects. | Use it for libraries or SDKs where every user-visible change should carry a reviewed release note fragment. | There is no public API or user-facing release stream yet. Changelog fragments would be ceremony without a package lifecycle. |
| [Commitizen](https://github.com/commitizen-tools/commitizen) - Conventional Commits helper, validator, and changelog/versioning tool. | Use it when a team wants commit-message enforcement and release automation to depend on commit semantics. | The base template avoids prescribing commit policy. Add it when the repository has agreed release and contribution rules. |
| [pre-commit](https://github.com/pre-commit/pre-commit) - widely used Git hook runner and hook ecosystem. | Use it if a downstream team prefers the standard Python hook runner or needs third-party hooks that assume pre-commit directly. | The template uses `prek` for hook execution while keeping a `.pre-commit-config.yaml`-compatible configuration shape. Installing both would split the hook story. |
| [codespell](https://github.com/codespell-project/codespell) - source-code spell checker. | Use it if a downstream project already standardizes on codespell dictionaries or suppressions. | The template already uses `typos`, which is fast and installed through `mise`. A second spell checker would add duplicate noise. |
| [Dev Containers](https://containers.dev/) - standard containerized development environment metadata for editors and remote workspaces. | Use it when contributors need a reproducible IDE/container setup, remote development, or onboarding that does not rely on host-installed tools. | The template already favors local reproducibility through `mise`, `uv`, and a Dockerfile. Dev Containers would add editor and container runtime assumptions. |
| [Docker Buildx](https://github.com/docker/buildx) - Docker BuildKit command plugin for repeatable and multi-platform image builds. | Use it when CI should prove the Dockerfile builds, publish images, or test BuildKit-specific features. | The template has a Dockerfile and Hadolint, but no container build or publishing policy. Previous build validation was also blocked by local Docker daemon availability. |
| [Task](https://github.com/go-task/task) - cross-platform task runner using `Taskfile.yml`. | Use it if a downstream team already runs Task across languages and wants Python commands to fit that convention. | `mise` already provides task orchestration and tool version management here, so Task would be another command surface for the same jobs. |
